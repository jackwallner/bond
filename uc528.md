# Bond — App Store Release Readiness Audit

_User-perspective audit to get Bond ready to **submit**, get **approved**, and **convert** browsers → downloaders → trial-starters → paying subscribers. Findings are grounded in the actual source; every item cites `file:line`._

**Verdict:** The app itself is well-built — purchase/restore handling is genuinely robust, the anon-first architecture is smart, empty states and gates are thoughtfully designed. But **it is not submittable today**, and even after the submission mechanics are fixed there are **2–3 things that will get it rejected**, plus a **structural conversion problem**: the app is solo-by-default but its paid value is couples-oriented, so a solo user who pays can't use the headline feature.

**Severity legend:** 🔴 blocker (rejection or can't submit) · 🟠 high (hurts approval odds or conversion materially) · 🟡 medium · ⚪ low

---

## 1. Can't submit yet (mechanics)

🔴 **No screenshots.** `fastlane/screenshots/` is empty (0 files). At least one 6.9" iPhone screenshot is mandatory to submit. Because Apple Watch is advertised in the title, subtitle, and description, and `BondWatch` is a real target, **if the Watch app ships in the build, Watch screenshots are also required.** This is also the single biggest download-conversion lever (see §4).

🔴 **Required metadata is blank in the repo** — verify each is actually set in App Store Connect, because these files are empty: `primary_category.txt` / `secondary_category.txt`, `en-US/privacy_url.txt`, `en-US/support_url.txt`, `copyright.txt`, `en-US/release_notes.txt`, and all of `review_information/*` (demo user/notes). A privacy policy URL and a category are required; release notes are required for a versioned release. The repo's pull script warns that ASC web edits get clobbered, so don't assume — confirm in ASC.

## 2. Will likely get rejected (approval)

🔴 **No in-app account deletion** (Guideline 5.1.1(v)). The app creates accounts — anonymous Supabase users on first launch (`SupabaseService.restoreSession` → `signInAnonymously`) and Apple-linked users on pairing (`signInWithApple`) — and persists user data (reminders, milestones, check-ins, couple). There is **only "Sign out" and "Unpair"** in `SettingsView.swift:21,46`; a grep across the app, migrations, and edge functions finds **no delete-account path at all**. This is one of the most consistently-enforced rejection reasons.
**Fix:** add a "Delete account" action in Settings that calls a Supabase edge function (service role) to delete the auth user + their rows, then resets to a fresh anonymous session.

🔴 / 🟠 **Background-location entitlement isn't justified by any feature** (Guideline 2.5.4). `Info.plist` declares `UIBackgroundModes → location` and `NSLocationAlwaysAndWhenInUseUsageDescription`, but:
- `LocationService.requestAuthorizationIfNeeded()` (the only thing that requests **Always**) is **never called anywhere** — grep confirms zero callers.
- The only auth actually requested is `requestWhenInUseAuthorization()` in `LocationService.currentLocation()` (one-shot, to drop a geofence pin).
- Geofence reminders fire via `UNLocationNotificationTrigger` (`NotificationScheduler.swift:138`), which needs only When-In-Use and does **not** require the `location` background mode. There is no `startUpdatingLocation`, no `allowsBackgroundLocationUpdates`, no CLLocationManager region monitoring.

Declaring persistent background location you don't use is a classic auto-reject.
**Fix:** remove `location` from `UIBackgroundModes`, remove `NSLocationAlwaysAndWhenInUseUsageDescription`, delete the unused `requestAlwaysAuthorization` path in `LocationService`. Keep only `NSLocationWhenInUseUsageDescription`.

🟠 **Fabricated statistic in the UI.** `ReminderEditorView.swift:167`: _"83% of Bond+ users report a healthier, happier relationship."_ There are no users yet — this is an unsubstantiated/false claim (Guideline 2.3.1, plus a real trust/legal liability).
**Fix:** delete it or replace with a non-numeric benefit line. The paywall's _"Join couples who use Bond+ to stay intentional together"_ (`PaywallView.swift:152`) is softer but still implies a userbase that doesn't exist — reword to aspirational ("Stay intentional, together") rather than social-proof framing.

🟠 **Reviewer can't exercise the paid/paired features.** `review_information/notes.txt` is empty. App Review will sign in anonymously (good — the app works without credentials), but **Daily Check-In requires a partner** (`DailyCheckInView.swift:19` → "For Couples Only" when solo) and Bond+ requires a purchase. Provide review notes that: (a) explain the auto-anonymous sign-in, (b) explain pairing needs two devices/accounts and how to generate+enter a code, (c) note Bond+ is testable via sandbox. Without this, reviewers often reject "couldn't access advertised features."

🟠 **"No third-party analytics" claim vs. reality.** The description says _"Privacy-first — no ads, no third-party analytics."_ The app ships RevenueCat (third-party; collects purchase/subscriber data) and Supabase (stores personal data). Make sure the **App Privacy nutrition label** declares Purchases + any linked identifiers, and soften the copy to something defensible like _"No ads. No data sold."_ A mismatch between the label, this copy, and the actual SDKs is a rejection vector.

🟡 **Universal-link domain mismatch.** Entitlement is `applinks:bond.jackwallner.com` (`Bond.entitlements`), but every legal/marketing link points to `jackwallner.com/bond/...` and the site lives there. Confirm `bond.jackwallner.com` actually serves a valid `apple-app-site-association` — otherwise invite links (`pairing.pendingInviteURL`) silently fail to deep-link, breaking the core pairing flow that users (and a reviewer) will try.

⚪ **`BondAppStoreID` is empty** in `Info.plist:18`, so the in-app "Write a review" deep link won't work. An app record already exists (per the asc-state scripts) — fill in the numeric ID now.

## 3. Conversion problem #1 — the solo/couples mismatch (most important)

Fix this before chasing screenshots. **Onboarding makes everyone solo** (`IntentSetupView` → `createSoloCouple`; pairing is opt-in and buried in Settings). Yet the paid value is couples-shaped:

- The paywall's **#1 listed benefit** is _"Daily Check-In, together"_ (`PaywallView.swift:42`), but Check-In is hard-gated to paired users (`DailyCheckInView.swift:19`). **A solo user who buys Bond+ literally cannot use the feature sold to them first.** That's both a conversion killer and an Apple "advertised feature unavailable" risk.
- The daily check-in is the strongest **habit loop** (the thing that earns a daily open and converts a trial), but it's double-locked: needs a partner _and_ needs Bond+. Free solo users — nearly everyone at first — never experience the hook that would make them want to pay.

**Recommendations:**
- Make the paywall adapt: for solo users, lead with benefits they can actually use (smart/location/surprise reminders, templates, insights) and frame Check-In as "when you pair."
- Surface pairing far more prominently (currently only in Settings and a card that appears once paired). Add a pairing nudge on the home screen for solo users.
- Consider letting free users answer a **solo** daily check-in (self-reflection prompt) so the habit forms before pairing/paying — then "answer together" becomes the upgrade.

## 4. Conversion #2 — listing → download (ASO)

- **Title** "Bond: Love Language Reminders" (30 chars) is solid and keyword-rich. Keep.
- **Subtitle** "Couples · Watch · Anniversary" wastes the highest-value 30 characters on a keyword list instead of a benefit hook. Try an outcome line, e.g. _"Remember the little things"_ or _"Show up for your partner."_ Move the stray keywords to the keyword field.
- **Keywords** has redundancy to reclaim: `relationship` overlaps the category; don't repeat words already in the title/subtitle (Apple indexes those). Add high-intent terms that are missing (e.g. `love,couple,reminder,anniversary reminder,date night,long distance`).
- **Promotional text is empty** — the one field changeable without review. Use it for a rotating hook / launch message.
- **Release notes empty** — required; write a first-version note.
- **Description above-the-fold**: first line is good; front-load the single most emotional benefit before the "…more" fold.
- **Screenshots (none exist):** for a zero-ratings launch, screenshots _are_ the social proof. Suggested first three frames with caption overlays: (1) "Never forget the little things" — home list with a partner reminder; (2) "Speak their love language" — reminder with love-language tag; (3) "One question a day, answered together" — the check-in reveal. Then Watch/widget, then insights. Captions matter more than raw UI.

## 5. Conversion #3 — download → trial start

The paywall (`PaywallView.swift`) is **strong**: single-screen, clear plans, honest savings/per-week math, correct 3.1.2 disclosures (auto-renew text, Restore, Terms/Privacy), trial-eligibility-aware CTA, good error/empty/loading states, robust pending-purchase handling. Keep it. Improvements:

- 🟠 **Purely gate-triggered** — no first-run or well-timed proactive paywall, so paywall _impressions_ (and trial starts) depend entirely on users bumping into a locked feature. Add one soft, dismissible paywall at a high-intent moment (e.g. after the 2nd–3rd reminder, or right after the pairing-success screen) to lift trial starts. Easy bail = Apple-compliant; the close button already exists.
- 🟡 Weak/empty social proof (see §2 — fix the 83% and "join couples" lines).
- 🟡 Gating placement is mostly good (preview-then-wall on Stats and Templates is exactly right). The home `TemplatesHomePitch` blurs the template peek for free users (fine), and Check-In shows the real question then walls the answer (good). Keep the preview pattern; just fix the solo problem in §3.

## 6. Conversion #4 — trial → paid (habit & retention)

🟠 **Notification permission primer is effectively dead on first launch — a real bug.** In `ReminderListView.swift` `.task`, `NotificationScheduler.reschedule(...)` (line 81) is called _before_ `maybeShowNotificationPrimer()` (line 85). But `reschedule` unconditionally calls `requestAuthorizationIfNeeded()` (`NotificationScheduler.swift:19`), which fires the **bare iOS permission dialog immediately** when status is `.notDetermined`. By the time `maybeShowNotificationPrimer` runs, status is no longer `.notDetermined`, so the primer's guard (`ReminderListView.swift:126`) fails and **the primer never shows.** New users get exactly the cold system prompt that `NotificationPrimerSheet` ("Bond is silent without your permission… that's the whole app") was built to prevent → more "Don't Allow" → no notifications → no retention → no trial conversion.
**Fix:** don't request authorization inside `reschedule`; show the primer first and request only after the user taps "Turn on notifications."

- 🟡 No in-app reinforcement near trial end / no win-back messaging. RevenueCat handles billing, but a gentle in-app "your trial ends soon — here's what you'll keep" moment would help.
- The review funnel (`ReviewPromptSheet` + `BondApp.swift`) is nicely gated to genuine positive moments (completing a reminder) with a feedback-vs-rate fork — good for store rating, which compounds download conversion. Leave it; ensure `BondAppStoreID` is set so the deep link works.
- Widgets + Watch complications are good retention surfaces — make sure at least one screenshot shows them, since they're a differentiator.

## 7. UX / smaller bugs

- 🟡 `OnboardingView.swift` (the standalone Sign-in-with-Apple screen) does not appear in `RootView`'s destination switch — looks like **dead code** (SiwA only actually appears via `AppleSignInPairingGate`). Confirm and delete to reduce confusion.
- 🟡 Importing a template pack of one-time reminders staggers them one **day** apart (`ReminderTemplatesView.swift:206`), so a 10-item pack spreads over 10 days starting tomorrow — may feel like "nothing happened." Consider staggering by hours, or clarify in the confirmation copy.
- ⚪ Location trigger requires tapping "Use current location" before saving, with only a small orange hint; the Save button stays enabled-looking (it's actually guarded). Minor polish.

## What's already done well (keep)

- Anon-first auth so the app is usable with zero sign-in friction; SiwA reserved for pairing/recovery (`SupabaseService`).
- Purchase robustness: pending/lag handling with sync + poll, restore offered in three places (Paywall, gate cards, Settings), error→Restore mapping, scenePhase refresh (`PurchasesService.swift`, `PaywallView.swift`, `BondApp.swift`).
- 3.1.2 paywall compliance (disclosures, Terms/Privacy/Restore present).
- Thoughtful empty states and preview-then-gate pattern on Stats/Templates.

## Recommended order

1. **Code fixes (in-repo):** account deletion; remove background-location/Always entitlements; delete the 83% claim; fix the notification-primer ordering. Clears the two hardest rejection risks + the worst retention bug.
2. **Conversion fix:** make the paywall solo-aware and surface pairing (§3); add one proactive paywall (§5).
3. **Submission assets:** screenshots (iPhone + Watch), categories, privacy/support URLs, review notes, release notes, promo text, `BondAppStoreID` — verify each in ASC.
4. **ASO polish:** subtitle + keywords rework (§4).
