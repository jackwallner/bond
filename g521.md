# Bond UX Analysis & Suggested Improvements (v521)

This report details key user experience (UX) friction points, bugs, and design opportunities identified within the **Bond** codebase. Each item links the user's immediate experience to the underlying technical implementation and offers concrete, actionable recommendations for improvement.

---

## 1. The Onboarding "Double-Loading" Router Glitch
* **User Experience:** When a user launches the app, they see a loading spinner. The screen briefly flashes the "Who do you want to show up for?" screen, but then immediately yanks them back to the loading spinner. A moment later, they are finally settled on the onboarding input screen or home dashboard. This makes the app launch feel unstable and unpolished.
* **Technical Context:** In [`RootView.swift`](file:///Users/jackwallner/bond/Bond/BondApp.swift#L125-L139), `currentDestination` determines the view routing. If `isTransitioning || !supabase.isAuthenticated`, it displays `.loading`. During `.task` at launch:
  1. `await supabase.bootstrap()` completes. If logged in, `isAuthenticated` becomes `true`, triggering a view re-render.
  2. Because `isTransitioning` is still `false` (the next line in the task hasn't executed yet or the state has not updated) and `pairing.coupleId` is `nil` (since `loadCouple()` hasn't run yet), the destination resolves to `.intentSetup`. The view starts animating.
  3. The next statement in the async task runs: `isTransitioning = true`. The router now resolves to `.loading` again, forcing the view back to the spinner.
  4. Once `pairing.loadCouple()` completes, it sets `isTransitioning = false`, animating to the final screen.
* **Suggested Improvement:** Do not evaluate routing destinations based on mid-flight variables. Instead, introduce a single state variable `isAppBootstrapped: Bool` that starts as `false` and is set to `true` only *after* both `bootstrap()` and `loadCouple()` have fully finished. Update the destination routing logic to check this single flag:
  ```swift
  if !isAppBootstrapped {
      return .loading
  }
  ```

## 2. Onboarding Gate: "Just Me" Setup Failure
* **User Experience:** A user launching the app for the first time selects "Just me" to explore the app. Instead of moving to the home screen, the setup fails instantly with a red error pop-up: `Could not find the function public.create_solo_couple(p_user) in the schema cache`. This blocks the user from using the app.
* **Technical Context:** In [`PairingService.swift`](file:///Users/jackwallner/bond/Bond/Services/PairingService.swift#L59-L72), `createSoloCouple()` executes a database RPC call:
  ```swift
  try await supabase.client
      .rpc("create_solo_couple", params: ["p_user": me.uuidString])
      .execute()
  ```
  PostgREST RPC resolution is strictly typed. If the database migration [`0002_solo_mode.sql`](file:///Users/jackwallner/bond/supabase/migrations/0002_solo_mode.sql#L14) defines the function as expecting a `uuid` type:
  ```sql
  create or replace function public.create_solo_couple(p_user uuid)
  ```
  PostgREST may fail to resolve the signature or throw cache mismatches if:
  1. The client passes it as a text string and PostgREST cannot automatically resolve the cast under local configuration.
  2. The local database migration has not been fully applied or granted execute permission for the `authenticated` role (which anonymous users occupy).
  3. PostgREST's schema cache is stale and has not been reloaded since the migration was run.
* **Suggested Improvement:** 
  - Ensure local development triggers `NOTIFY pgrst, 'reload schema';` when migrations run.
  - Verify parameter signatures. If PostgREST fails to automatically coerce text to uuid, overload the function in PostgreSQL to accept a text input and cast internally, or pass raw types directly from the Swift SDK.

## 3. Account Promotion Data Loss (Anonymous to Apple Sign-In)
* **User Experience:** A user uses Bond in "Just me" (anonymous) mode, setting up reminders, choosing love languages, and entering relationship details. Later, they want to pair with their partner. The app prompts them to Sign in with Apple to protect their account. They sign in, but all their previous reminders and notes vanish, sending them back to the onboarding screen.
* **Technical Context:** In [`PairingView.swift`](file:///Users/jackwallner/bond/Bond/Features/Pairing/PairingView.swift#L332-L352), the Apple Sign-In handler upgrades the user using `supabase.signInWithApple(...)` which calls:
  ```swift
  let session = try await client.auth.signInWithIdToken(
      credentials: .init(provider: .apple, idToken: idToken, nonce: nonce)
  )
  ```
  This signs the user into a completely *new* Supabase auth identity (new UUID) rather than linking their current anonymous user record in-place. Because their reminders and milestones are linked via database relationships to the anonymous user ID, those records are orphaned and lost to the user.
* **Suggested Improvement:** Use the GoTrue account linking API to associate the Apple credential with the *current* anonymous session instead of creating a brand new session. Use `supabase.client.auth.linkIdentity` (or the equivalent identity linking method in the Supabase Swift library) to promote the anonymous account to a permanent Apple-linked user without changing the database user ID (`auth.uid()`).

## 4. Universal Link Bypasses Account Recovery Gate
* **User Experience:** An anonymous user receives an invite link from their partner (e.g. `https://bond.jackwallner.com/pair/ABCDEF`). They tap it. The app opens and immediately pairs them. They are delighted by the instant pairing but remain completely anonymous. A week later, they delete the app or get a new phone; their account is lost forever, leaving their partner permanently paired with a dead anonymous account.
* **Technical Context:** In [`PairingService.swift`](file:///Users/jackwallner/bond/Bond/Services/PairingService.swift#L162-L169), `handleIncomingURL` immediately calls `consumeInviteCode(...)` upon detecting a pairing path. This executes the `consume_invite_code` RPC. The client does not check if the user is anonymous before completing the pairing. While standard pairing in [`PairingView.swift`](file:///Users/jackwallner/bond/Bond/Features/Pairing/PairingView.swift#L22) locks anonymous users behind `AppleSignInPairingGate`, the universal link path bypasses this control entirely.
* **Suggested Improvement:** In `handleIncomingURL(_:)`, check `supabase.isAnonymous`. If the user is anonymous, store the invite code in a temporary variable, redirect the user to the `AppleSignInPairingGate`, and execute the `consumeInviteCode` RPC only *after* Apple Sign-in completes successfully.

## 5. Editor UX: Invisible Form Validation Failures
* **User Experience:** A user adds a new reminder and selects the "Location" trigger. They forget to tap "Use current location". The "Save" button remains active. Tapping it starts a loading spinner, but the screen does not close. If they scroll down, they might see a red validation error at the bottom. The same issue occurs if they set a "Random Window" trigger where the end time is earlier than the start time.
* **Technical Context:** The "Save" button disabled state in [`ReminderEditorView.swift`](file:///Users/jackwallner/bond/Bond/Features/ReminderEditor/ReminderEditorView.swift#L119) is only bound to the title text:
  ```swift
  .disabled(title.trimmingCharacters(in: .whitespaces).isEmpty || isSaving)
  ```
  Validation for locations and date windows is evaluated asynchronously inside the `save()` method, setting `errorMessage`.
* **Suggested Improvement:** Bind the Save button's disabled state to form validity:
  ```swift
  private var isFormValid: Bool {
      guard !title.trimmingCharacters(in: .whitespaces).isEmpty else { return false }
      switch triggerKind {
      case .location:
          return geofenceConfigured
      case .randomWindow:
          return windowEnd > windowStart
      default:
          return true
      }
  }
  ```
  Additionally, display inline warnings in yellow/red directly beneath the location capture button or window date pickers so the user understands why the form is incomplete before tapping Save.

## 6. Template Import Latency: Sequential Network Spam
* **User Experience:** A user is excited to try relationship templates. They tap "Add 7 reminders" inside a template group. The button spinner freezes for 3 to 4 seconds, locking the screen and making the app feel slow and laggy.
* **Technical Context:** In [`ReminderTemplatesView.swift`](file:///Users/jackwallner/bond/Bond/Features/Templates/ReminderTemplatesView.swift#L146-L181), `createAll()` loops through templates and calls `repo.upsert(reminder)` sequentially. Inside [`ReminderRepository.swift`](file:///Users/jackwallner/bond/Bond/Services/ReminderRepository.swift#L47-L57), `upsert` inserts the row and immediately awaits `refresh()` (fetching all reminders). For 7 reminders, the app sends 7 independent HTTP updates followed by 7 complete list queries sequentially over the network—resulting in 14 sequential API calls.
* **Suggested Improvement:** Add a bulk insert method to `ReminderRepository` that takes an array `[ReminderDTO]`. Execute a single bulk database insert:
  ```swift
  func bulkInsert(_ reminders: [ReminderDTO]) async throws {
      try await supabase.client
          .from("reminders")
          .insert(reminders)
          .execute()
      await refresh()
  }
  ```
  This reduces the network roundtrips from 14 to 2, importing templates instantly.

## 7. Templates Schedule Flooding (Notification Fatigue)
* **User Experience:** When a template group is imported, the user's phone receives 7 notifications simultaneously in exactly 1 hour. This causes notification fatigue and frustrates the user.
* **Technical Context:** In [`ReminderTemplatesView.swift`](file:///Users/jackwallner/bond/Bond/Features/Templates/ReminderTemplatesView.swift#L165), all imported templates have their initial trigger date hardcoded to:
  ```swift
  fireAt: Date().addingTimeInterval(60 * 60)
  ```
* **Suggested Improvement:** Stagger the schedule of imported reminders. Use spacing logic in templates to set their dates over subsequent days or weeks (e.g., Day 1 at 9:00 AM, Day 2 at 6:00 PM, Day 4 at 12:00 PM) based on template metadata. Alternatively, let users customize the schedule in a preview list before saving.

## 8. Permanent Loss of Quick Inspiration Chips
* **User Experience:** On first launch, the user sees helpful "Starter Chips" (e.g., "Bring home flowers", "Phone away during dinner") that make it easy to add reminders. However, as soon as they add a single reminder, the chips vanish forever. If they run out of ideas later, they have to navigate through menus to find inspiration.
* **Technical Context:** In [`ReminderListView.swift`](file:///Users/jackwallner/bond/Bond/Features/ReminderList/ReminderListView.swift#L29-L40), `EmptyRemindersView` is only shown when `visibleReminders.isEmpty`.
* **Suggested Improvement:** Reposition the Starter Chips to remain accessible. They could be displayed as a horizontally scrollable suggestions bar at the bottom of the reminder list, or integrated as quick-select ideas directly inside the new reminder creation sheet.

## 9. Apple Watch Dictation Target Mismatch
* **User Experience:** A user dictates a reminder on their Apple Watch: "Pick up favorite ice cream." They expect this reminder to target their partner, or let them choose. However, it always targets the user themselves, defeating the purpose of relationship-centered prompts.
* **Technical Context:** In [`WatchConnectivityBridge.swift`](file:///Users/jackwallner/bond/Bond/Services/WatchConnectivityBridge.swift#L82-L99), the received watch reminder payload defaults the target ID to the user:
  ```swift
  let reminder = ReminderDTO(
      ...
      authorId: me,
      targetId: me,
      ...
  )
  ```
* **Suggested Improvement:** If the user is paired, default the dictated reminder's target to the partner (`targetId: partnerId`). Better yet, add a simple toggle on the watch interface in [`DictateView.swift`](file:///Users/jackwallner/bond/BondWatch/DictateView.swift) to choose the recipient.

## 10. Daily Check-In: Plain Reveal Experience
* **User Experience:** The climax of the daily connection flow occurs when both partners have submitted their check-in answers, revealing each other's notes. Currently, this moment is a plain fade-in, making the interaction feel dry and anticlimactic.
* **Technical Context:** In [`DailyCheckInView.swift`](file:///Users/jackwallner/bond/Bond/Features/CheckIn/DailyCheckInView.swift#L141-L157), the partner's card transitions using a standard `.opacity` animation.
* **Suggested Improvement:** 
  - **3D Card Flip:** Implement a realistic 3D Y-axis card flip animation (`.rotation3DEffect`) where the user taps to flip the sealed envelope card, showing the partner's answer on the reverse.
  - **Scratch-to-Reveal:** Provide an interactive canvas scratch-off effect where users swipe their finger to "scrape" away a foil layer, triggering minor haptic clicks (`UIImpactFeedbackGenerator(style: .light)`) as they reveal the text.
  - **Haptic Feedback:** Integrate subtle tactile feedback on form submissions, successful pairing, and milestone completion.
