# Bond UX Analysis — Suggested Improvements (c521)

Read-only review of the Bond iOS + Watch codebase, written from **how someone actually uses the app** — what they expect, where they get stuck, and what should change. No code was modified for this document.

**Scope:** Current working tree (including in-progress fixes visible in local changes). Where an issue is already addressed locally, it is noted under [Recently improved](#recently-improved-in-working-tree) so this list stays focused on what users still feel.

**Related:** `g521.md` (earlier pass), `docs/MVP_TRIAGE.md` (engineering triage), `claude-design/PRODUCT.md` (intended product shape).

## Recently improved (in working tree)

These were real user-facing problems; the current branch largely fixes them. Keep them in mind for regression testing, not as open work:

- **Launch flicker** — Single `isAppBootstrapped` gate in `RootView` avoids intent-setup → spinner → home flashes (`BondApp.swift`).
- **Anonymous → Apple upgrade** — `linkIdentityWithIdToken` preserves `auth.uid()` and data (`SupabaseService.swift`).
- **Invite link before sign-in** — Anonymous users defer pairing and see `AppleSignInPairingGate` via sheet (`PairingService.handleIncomingURL`, `RootView`).
- **Reminder editor validation** — Save disabled until title + location/window rules pass; inline orange hints (`ReminderEditorView`).
- **Starter ideas** — Chips live in the editor (“Need an idea?”), not only on empty list (`ReminderEditorView`, `EmptyRemindersView`).
- **Template import** — Bulk insert + staggered `fireAt` by day (`ReminderTemplatesView`, `ReminderRepository.bulkInsert`).
- **Check-in reveal** — Tap-to-flip card with haptics (`DailyCheckInView`, `FlipRevealCard`).
- **Watch dictation** — Partner/me picker; phone replies with real success/failure (`DictateView`, `WatchConnectivityBridge`).
- **Daily check-in question sync** — Deterministic couple seed (`DailyCheckInService.stableSeed`).

## 1. First launch & onboarding

### 1.1 Solo users are asked for “someone else” before they can use the app

**User experience:** Someone downloads Bond for self-care or journaling. They expect to land on *their* reminders. Instead they must name another person, commit to “showing up for [name],” and pick that person’s love language — even if they are not pairing yet.

**What happens today:** `IntentSetupView` is the only path when `coupleId == nil`. Copy is partner-centric (“Who do you want to show up for?”). Finishing calls `createSoloCouple()`; there is no “this is for me” branch.

**Suggested change:** Offer an explicit path: “Mostly for me” vs “For someone I care about.” Self path: shorter setup (optional display name, default love language, focus areas about *self*). Partner path: keep today’s flow. Align copy with `PRODUCT.md` solo positioning without forcing a fictional partner.

---

### 1.2 First-time setup can fail with a database error message

**User experience:** On the last onboarding step they tap “Start using Bond” and see a red error like *Could not find the function public.create_solo_couple(p_user) in the schema cache*. The app feels broken; they cannot reach the home screen.

**What happens today:** `IntentSetupView.finish()` → `PairingService.createSoloCouple()` → RPC `create_solo_couple`. Failure surfaces `pairing.lastError` as raw PostgREST text. Migration `0006_solo_couple_text_overload.sql` (text overload + schema reload) may not be applied on the environment they hit.

**Suggested change:** For users: human copy (“We couldn’t finish setup — try again in a moment”) plus retry, not schema jargon. For ops: ensure migration 0006 is deployed everywhere TestFlight/production talk to. Optional: client-side retry once after a short delay if error matches stale cache.

---

### 1.3 Onboarding “their name” does not become the real partner’s name after pairing

**User experience:** They carefully enter “Sarah” during setup. After pairing, the list still says “Your partner” or a blank name until the partner sets a profile — the name they typed feels ignored.

**What happens today:** `OnboardingPreferences` (UserDefaults) stores partner name, love language, and focus areas locally. Comments in code say paired users use server `profiles` for shared fields; onboarding hints stay private and are not pushed on pair.

**Suggested change:** After pairing, if partner `displayName` is empty, pre-fill partner profile setup with onboarding name (with consent: “Use the name you entered for them?”). Surface partner name in home chrome (`ReminderListView` already has `partnerName` — use it in nav subtitle or section headers consistently).

---

### 1.4 Notification primer never runs for solo users

**User experience:** A solo user creates their first reminder and gets the iOS system notification alert with no context — easy to tap “Don’t Allow,” then wonder why reminders never fire.

**What happens today:** `ReminderListView.maybeShowNotificationPrimer()` returns early when `pairing.solo`. Primer only runs for paired users who have not been asked yet. Solo users still trigger `NotificationScheduler.requestAuthorizationIfNeeded()` on first save via `reschedule()`.

**Suggested change:** Show `NotificationPrimerSheet` for solo users too — ideally before the first save (e.g. when opening the editor or after first successful reminder), with solo-specific copy (“Bond nudges *you* at the right time”). Same sheet, different subtitle.

## 2. Daily reminders (core loop)

### 2.1 Reminders “for my partner” often never notify the partner

**User experience:** They create “Send flowers Friday” targeted at their partner, assuming the partner’s phone will buzz. The partner never hears anything. The author thinks the app is unreliable.

**What happens today:** `NotificationScheduler.reschedule` only schedules local notifications where `reminder.targetId == selfId`. Partner-targeted rows rely on push via `send-push` and `profiles.apns_token`. Client does not register for remote notifications or persist APNs token (`docs/MVP_TRIAGE.md` B3).

**Suggested change:** Until push ships: set expectations in the editor (“Your partner will see this in Bond when they open the app — notifications for them are coming soon”) or temporarily disable/hide partner targeting. When push ships: register token on launch, write to profile, and verify end-to-end on two physical devices.

---

### 2.2 Tapping a notification does not open the relevant reminder

**User experience:** They tap a Bond notification expecting to land on that reminder or mark it done. The app opens to the generic list (or last state).

**What happens today:** `userInfo` includes `reminder_id`, but there is no `UNUserNotificationCenterDelegate` to handle taps and deep-link into `ReminderEditorView` for that id.

**Suggested change:** Handle notification response → resolve reminder id → present editor or scroll list to row. Reduces friction for the core “nudge → act” loop.

---

### 2.3 “Past due” vs “Handled” is confusing

**User experience:** A one-time reminder fires yesterday. It sits under **Past due** next to active items. They swipe **Done** and it moves to **Handled** but still looks like a normal row — the list feels cluttered, like a todo app that never clears.

**What happens today:** `isPastDue` only applies to one-time triggers with `fireAt < now`. Marking done creates a `reminder_events` row; item moves to Handled section but is not archived or hidden from active scheduling. Underlying reminder row may still be `scheduled`.

**Suggested change:** When user marks done on a one-time (or exhausted random-window) reminder, auto-archive or move to a compact “Recently handled” with clearer visual distinction (muted style, no time emphasis). Optionally auto-dismiss past-due one-times after Done. Frame copy as “logged” not “incomplete task.”

---

### 2.4 Sign out can trap the user on a loading spinner

**User experience:** From Settings they sign out to test another account or “start over.” The screen shows an endless spinner; force-quit is the only escape.

**What happens today:** `SettingsView` calls `supabase.signOut()` which clears `currentUserId`. `RootView` routes to `.loading` when `!supabase.isAuthenticated`. `.task` bootstrap does not re-run on sign-out; nothing calls `signInAnonymously()` again.

**Suggested change:** After sign-out, immediately start a fresh anonymous session (or return to a dedicated “signed out” screen with one CTA: “Continue”). Treat sign-out as “new guest session,” not terminal loading.

---

### 2.5 Partner’s new reminders may not appear until app restart (solo → paired)

**User experience:** They pair successfully and expect live updates when their partner adds a reminder. Their list stays stale until they kill and reopen the app.

**What happens today:** `ReminderRepository.subscribeRealtime()` subscribes once per channel and early-returns if `realtimeChannel != nil`. Solo session may have subscribed with an old/missing couple filter; pairing does not tear down and resubscribe.

**Suggested change:** On `coupleId` change (pair, unpair, sign-in refresh), `unsubscribeRealtime()` then `subscribeRealtime()`. User-visible: optional “Updated just now” after realtime refresh.

---

### 2.6 Premium features are hard to discover

**User experience:** Product docs describe tabs: Reminders, Check-In, Milestones, Insights. In the app they only see **Bond** (reminders) and an ellipsis **More** menu. Check-in appears as a card only when paired; Milestones and Insights are buried.

**What happens today:** `RootView` home is only `ReminderListView`. `BondMoreView` lists Check-In, Milestones, Insights, Settings. No tab bar.

**Suggested change:** Add a lightweight tab bar (Reminders | More) or bottom accessory for paired users: Check-In badge when partner has answered. At minimum, first paired session: one-time coach mark pointing to More / Check-In. Match `PRODUCT.md` information architecture so premium value is discoverable without hunting.

## 3. Pairing & account recovery

### 3.1 Apple Sign-In with an existing Apple ID can silently drop anonymous data

**User experience:** They used Bond anonymously, built up reminders, then sign in with Apple using an account that already exists on another device. Their list empties; they think the app deleted everything.

**What happens today:** `signInWithApple` tries `linkIdentityWithIdToken` for anonymous users; on failure it falls back to `signInWithIdToken`, which signs into a *different* user id. Comment in code acknowledges anonymous data loss in that case; UI does not warn beforehand.

**Suggested change:** Before fallback, explain: “This Apple ID already has a Bond account. Continuing will switch accounts and leave this device’s current reminders behind.” Offer “Switch account” vs “Stay on this device.” Power users need informed consent, not surprise loss.

---

### 3.2 Anonymous users have no “reset” or sign-out in Settings

**User experience:** They want to clear local state or hand the phone to someone else. Settings has no sign-out for anonymous accounts — they feel stuck in one guest identity.

**What happens today:** Sign out button only appears when `!supabase.isAnonymous`. Anonymous users only see Pairing, Premium, Theme, Notifications, legal links.

**Suggested change:** “Start over” for anonymous users: sign out + new anonymous session + clear local onboarding prefs (with confirmation). Same outcome as QA needing two fresh accounts on one phone.

---

### 3.3 Pairing success screen auto-advances without an explicit Continue (most users)

**User experience:** Hearts animate together, then the screen vanishes ~1.8s later. They may miss the “You & [name]” moment or want to screenshot/share.

**What happens today:** `PairingSuccessView` auto-calls `onDone()` after animation unless Reduce Motion is on (then they get an explicit Continue button).

**Suggested change:** Always show Continue (or tap anywhere). Keep animation; don’t auto-dismiss. Optional light haptic at settle — pairing is emotionally meaningful in this product.

## 4. Templates & premium

### 4.1 Template import reports success even when nothing was added

**User experience:** They tap “Add 7 reminders,” see “Templates added!”, but the list is unchanged.

**What happens today:** `TemplateGroupDetailView.createAll()` sets `showConfirmation = true` after the `do/catch` regardless of success; `createdCount` is 0 on failure but the alert copy does not reflect that.

**Suggested change:** Only show success when `createdCount > 0`. On failure, inline error (“Couldn’t add templates — check connection and try again”) with retry. Never celebrate zero inserts.

---

### 4.2 Template import targets the partner by default when paired

**User experience:** They browse “New Parents” to remind *themselves* what to do as a parent. All seven reminders target the partner because `targetId` resolves to `partnerId` when not solo.

**What happens today:** `createAll()` sets `targetId: pairing.solo ? me : (partnerId ?? me)`.

**Suggested change:** Pre-import sheet: “Who are these reminders for?” (Me / Partner / Mix). Or default to **me** with clear toggle — authors often template for their own behavior first.

---

### 4.3 Milestones may not match premium positioning

**User experience:** Marketing and `PRODUCT.md` say Milestones are Premium. They open Milestones from More and get full create/list UX with no gate — feels inconsistent if they later hit a paywall elsewhere.

**What happens today:** `MilestonesView` has no `store.isPremium` check (unlike Templates, Check-In, Insights).

**Suggested change:** Align with product: either gate with `BondGatePreview` like other premium surfaces, or update product copy if Milestones are intentionally free for MVP. Inconsistency erodes trust in what “Premium” means.

---

### 4.4 Free users see the check-in question but cannot participate

**User experience:** Paired, non-premium user opens Check-In, reads today’s intimate question, then hits a paywall to answer. Feels like a teaser ad inside an emotional feature.

**What happens today:** `teaserContent` loads and displays the real `questionCard` plus `BondCheckInUnlockCard`.

**Suggested change:** Either blur/hide question text for free users (show category only: “Today’s question is about appreciation”) or offer a limited free tier (e.g. one check-in per week). Reading the full question without answering can feel worse than not showing it.

## 5. Location, random, and editor edge cases

### 5.1 Location reminders may not fire when the app is in the background

**User experience:** They set “When I get home, hug her,” grant location once, and expect it to work like Life360. It never fires unless they happen to have the app open.

**What happens today:** `LocationService.currentLocation()` calls `requestWhenInUseAuthorization()` for capture. `requestAuthorizationIfNeeded()` (Always) exists but is not called from the editor before saving a geofence trigger. `NotificationScheduler` uses `UNLocationNotificationTrigger`, which needs appropriate authorization and region monitoring.

**Suggested change:** When user selects Location trigger, walk them through Always permission with plain language (“Bond needs background location to nudge you when you arrive”). Block save until Always (or downgrade feature with honest “only works while app is open” mode).

---

### 5.2 Random “surprise” windows feel like one lottery ticket

**User experience:** They set a weekly random window expecting ongoing surprises. One notification fires; the window never rolls again.

**What happens today:** On save, one random `fireAt` is picked inside the window and stored as one-time-style scheduling. No re-roll after fire.

**Suggested change:** Either re-randomize after each fire (and say so in UI) or rename copy to “One surprise between …” so expectations match. Premium users paying for “surprise” expect recurrence unless told otherwise.

---

### 5.3 Watch dictation always schedules ~1 hour out

**User experience:** On the watch they dictate “Tonight pick up dinner.” They assume “tonight” means tonight, not “in about an hour” from send time.

**What happens today:** `WatchPayload.CreateReminder.scheduledOffsetSeconds` drives `fireAt`; watch copy says “Sent — will fire in ~1h.”

**Suggested change:** Short-term: accurate copy (“Will remind you in about an hour”). Better: quick presets on watch (In 1h / Tonight / Tomorrow morning) or honor Siri/time phrases if feasible.

---

## 6. Watch & widgets

### 6.1 Watch has no way to see or complete existing reminders

**User experience:** They wear the watch to stay present, not pull out the phone. They dictate one reminder but cannot glance at what’s upcoming or mark Done.

**What happens today:** Watch app is dictation-only (`DictateView`). No list, no complications beyond what widgets on phone provide.

**Suggested change:** Acceptable for MVP if positioned honestly in App Store/watch marketing (“Dictate to phone”). Post-MVP: top 3 upcoming “for me” on watch + Done action mirroring phone swipe — matches “small act on cue” without full editor.

---

### 6.2 Background watch send may still show stale success (application context path)

**User experience:** Phone is locked in pocket; watch says “Sent.” Reminder never appears.

**What happens today:** Reply handler path returns real errors. `didReceiveApplicationContext` uses `handleCreatePayload` without reply — watch may not get failure feedback on that path.

**Suggested change:** Unify both WCSession paths to report failure back to watch, or show “Queued — open Bond on phone to confirm” when using application context.

## 7. Trust, settings, and copy

### 7.1 Stale pairing errors can linger across screens

**User experience:** Invite failed once yesterday. Today they retry successfully but still see yesterday’s red error at the bottom of Pairing or onboarding.

**What happens today:** `PairingService.lastError` is cleared on some success paths but not universally before new operations; `IntentSetupView` shows `pairing.lastError` without clearing on appear.

**Suggested change:** Clear `lastError` at the start of every user-initiated pairing action and on view appear. Errors should describe the *latest* attempt only.

---

### 7.2 Check-in and milestone failures are invisible in the UI

**User experience:** They submit a check-in answer; nothing happens. They don’t know if it’s loading, failed, or succeeded.

**What happens today:** `DailyCheckInService.lastError` is set on failure but `DailyCheckInView` does not bind it. Similar pattern for other services’ `lastError` fields.

**Suggested change:** Surface errors inline under Submit (BondInlineError) and on load failure replace spinner with retry. Same pattern for Milestones save/load.

---

### 7.3 `FilteredEmptyView` suggests a filter UX that does not exist

**User experience:** N/A today — component is unused.

**What happens today:** `FilteredEmptyView` (“Nothing for you right now”) is defined in `EmptyRemindersView.swift` but never inserted in `ReminderListView` (older filter model removed or never shipped).

**Suggested change:** Either wire up partner/self filtering with this empty state or delete dead UI to avoid future confusion. If the list is intentionally unified (author + partner sections), remove unused view.

## 9. Suggested validation (no code)

When implementing fixes, validate as a user would:

1. **Two phones, one couple** — Partner-targeted reminder → partner device notifies (or explicit in-app message if push deferred).
2. **Sign out → continue** — Never stuck on spinner; can use app again within 5 seconds.
3. **Fresh install solo** — Complete onboarding without RPC errors on production Supabase.
4. **Tap notification** — Lands on correct reminder.
5. **Import template** — Airplane mode → failure message, not success.
6. **Pair after solo** — Partner adds reminder; other phone updates without restart.
7. **Watch** — Dictate for partner with phone locked → reminder appears or watch shows actionable error.
